## 1. 왜 개편했는가 — 증거 체인 (절대 잊지 말 것)

이 개편은 직감이 아니라 **누적된 증거**에 기반한다. 세 가지 증거가 한 방향을 가리켰다.

### 증거 1: 키워드 데이터 (가장 강력 — 이미 수집 완료)
- 살아있는 키워드: **"장기요양 기록"(46.2)**, **"방문요양 평가 준비"(22.0)**, "재가급여"(46.2)
- 전멸한 키워드: "환수"(0), "현지조사"(0), "부당청구"(0), "행정처분"(0) — 전부 0
- **해석:** 센터장은 *당한 뒤*(환수·조사)엔 검색 안 함. *기록을 쓰는 중*(기록·평가 준비)에 검색함.
- **결론:** 검색 시점 = 작성 시점 = 불안의 순간. 제품은 그 순간에 있어야 한다.

### 증거 2: 네이버카페 자료 (실무자 육성)
- "뭘 적어야 할지 모른다"
- "작성해야 하는 서류가 너무 많다"
- "메모지·카톡·엑셀로 따로 관리한다" (케어포 밖에서)
- "평가 때 무작위 질문에 긴장한다"
- "준비했는데도 감점당한다"
- **해석:** 케어포가 빈 양식은 주지만 "뭘 어떻게 써야 하는지"는 안 알려줌. 그래서 메모지·카톡으로 보완.

### 증거 3: 케어포/엔젤케어의 구조적 허점 (시장 검증)
- 케어포도 **수동 입력**이다 (RFID·청구파일 생성만 자동, 나머지 전부 수동).
- 케어포는 기록을 **만드는** 도구. "이 칸에 뭘 써야 평가에서 안 깎이는지"는 안 알려줌.
- 케어포를 쓰는데도 **1,342곳이 부당청구 적발**, 수많은 센터가 감점.
- **해석:** "작성 가이드"는 케어포가 비운 칸. 케어포가 따라오기 어려운 영역(ERP는 평가 컨설팅을 안 함).

### 핵심 통찰 (이 사업의 척추)
```
케어포 = 양식을 준다 (빈칸)
컨설턴트 = 1회 20~50만원에 "이렇게 쓰세요" 알려준다
우리 = 컨설턴트의 지식을, 작성하는 순간, 항목별로, 자동으로 (월 99,000원)
```
경쟁 상대는 케어포(월 3~8만)가 아니라 **컨설턴트(1회 20~50만)**. 가격 비교 대상이 바뀐다.

## 2. 제품의 3개 층 (하나의 흐름)

```
[Level 3] 작성 가이드  "이 칸에 뭘 어떻게 써야 하나?"  ← 가장 깊은 고통, 가장 강한 차별
    ↓
[Level 2] 연쇄 알림    "A를 썼으면 B도 써야 한다"        ← 누락의 진짜 원인 (★핵심)
    ↓
[Level 1] 검증         "이 기록들이 서로 맞나?"          ← 이미 만든 엔진(14개 규칙)
```

### ★ Level 2 "연쇄 알림"이 제품의 척추 (2026-06-19 확정)

센터장이 까먹는 건 *하나의 서류*가 아니라 **"A를 쓰면 따라와야 하는 B"**다.
케어포는 A를 저장하고 끝낸다. B가 따라와야 한다고 알려주지 않는다.
이게 누락의 진짜 원인이고, 우리 14개 규칙이 바로 이 연쇄를 담고 있다.

**연쇄 예시 (우리 엔진이 이미 아는 것):**
- 상태 악화 기록 → 급여제공계획 변경 검토 (Y05)
- 결과평가 실시 → 30일 내 계획 재작성 (Y11)
- 급여(시간·제공자·횟수) 변경 → 계획서 갱신 (Y18)
- 욕구사정 실시 → 급여제공계획 반영 (Y01)

**핵심 재정의:** 우리 엔진은 "검증 엔진"이 아니라 처음부터 **"연쇄 추적 엔진"**이었다.
"검증"이라는 틀로 잘못 보고 있었을 뿐. 같은 엔진, 다른 진입점.

**작동 방식:**
```
센터장이 "상태변화 기록 작성" 선택
 → 조합형 가이드로 작성 (예: "식사량 감소...")
 → [복사] → 케어포에 붙여넣기
 → ★ 앱 즉시 알림: "상태가 악화로 기록됨.
    → 이번 주 안에 '급여제공계획 변경 검토'도 하셔야 합니다.
    → 안 하면 Y05·Y11 연쇄 감점 위험."
 → "급여제공계획 변경" 작성으로 자연스럽게 연결
```

세 층은 따로가 아니라 하나의 흐름으로 연결된다:
```
"이번 달 김영순님 상태변화기록 작성하셔야 합니다" (연쇄 알림)
   ↓
"상태변화기록엔 신체·정서·식사·수면·조치 중 최소 1개를 구체적으로.
 예시: '식사량 1/3 감소, 보호자 유선 안내, 다음 방문 시 재확인'" (작성 가이드)
   ↓
"이 기록이 악화면 → 급여제공계획 변경도 하셔야 합니다" (연쇄 알림)
```

**무게중심:** Level 2(연쇄 알림) + Level 3(작성 가이드)이 앞. Level 1(검증)은 뒤.
기존 앱은 Level 1만 있었음. 이제 연쇄 알림이 누락을 원천 차단하는 핵심.

## 3. 엔진은 살아있다 (버리는 것 아님)

기존에 만든 14개 규칙(R01~R04, R07, Y01~Y11, Y18)은 **그대로 자산**이다.
- 검증 로직(Level 1) = 이미 완성
- 각 규칙의 "근거"(공단 매뉴얼 조문) = 작성 가이드(Level 3)의 원료
- 각 규칙의 "주기"(회계연도/반기/30일) = 누락 예측(Level 2)의 원료

즉 **엔진은 그대로 두고, 표현·시점·진입점을 바꾼다.** "검증 결과 화면" → "작성 안내 화면".

### 공식 기록 흐름 체인 (연쇄 알림의 근거 — 법적 순서)
```
상담일지(업무수행일지)
  → 욕구사정
  → 위험도평가(낙상·욕창·인지)
  → 급여제공계획 수립 (★허브)
  → 급여제공계획 설명·서명·통보
  → 장기요양급여 제공기록지 작성
  → 급여제공 결과평가
  → 계획 재작성 또는 유지 근거
  → (루프)
```
이 체인이 우리 연쇄 알림(Level 2)의 법적 근거.
현장에서는 이 순서대로 안 일어남(비순차) → 그래서 더 누락됨 → 우리 앱이 필요.

### 우리 규칙이 잡는 연쇄 매핑 (현재 엔진)
| 규칙 | 연쇄 | 의미 |
|---|---|---|
| Y01 | 욕구사정 → 계획 반영 | 사정했으면 계획에 반영됐나 |
| Y05 | 상태 악화 → 계획 변경/상담 | 악화됐으면 조치했나 |
| Y11 | 결과평가 → 30일 내 계획 재작성 | 평가했으면 재작성했나 |
| Y18 | 급여 변경 → 계획서 갱신 | 변경됐으면 갱신했나 |

### ★ 신규 수급자 예외 규정 (미반영 → 반영 필요)
- 신규 수급자는 급여개시일이 속한 반기에는 결과평가 확인 대상에서 **제외**될 수 있음.
- 현재 엔진(Y11)에 이 예외가 없음 → **거짓양성 위험**
- 수급자 등록 시 "신규/기존" 구분 + 급여개시일 입력 → Y11 judge에 예외 로직 추가 필요
- **구현 시점:** 수급자 등록 기능 구현할 때 함께.

### 현지조사 교차 확인 서류 (미래 로드맵)
현지조사 시 단일 기록이 아니라 교차 확인함:
본인부담금 수납대장 ↔ 통장 입금내역 ↔ CMS
근무상황부 ↔ 출근부 ↔ 급여대장
→ 이건 "현지조사 준비팩" 기능으로 MVP 이후 구현.
→ 현재 SSOT §6-E(제품 로드맵)의 "그룹 I: 현지조사 준비팩"에 이미 예약됨.

### ★ 거짓양성 제거 규칙 5개 (장기요양.md 통합, 2026-06-19)
신뢰 = 99,000원의 본체. 틀린 경고 하나가 신뢰를 무너뜨림. 이 5개를 엔진에 반영:
1. **입원·외박 중 제외:** 수급자 입원·장기외박 기간은 서비스 불가 → 그 기간 규칙 비활성.
   (수급자 등록 시 "입원/외박 기간" 입력)
2. **공휴일 익일 규정:** 14일·30일 기한 마지막 날이 토/공휴일이면 다음 영업일까지 인정.
3. **신규 직원 상담 예외:** 신규 직원은 급여제공 시작 월의 다음 분기부터 상담 → 입사 분기 경고 금지.
4. **서명 예외 (비고 확인):** 관절구축 등 서명 불가 시 비고란 사유 있으면 인정 → 빈칸이라고 바로 경고 말고 비고 먼저 확인.
5. **비대면 통보 인정:** 우편·SNS 전송 후 유선 안내 기록 있으면 충족 (D02 강화).
→ 구현 시점: 수급자 등록 + 엔진 개편 시 함께.

## 3-A. 작성 가이드(Level 3) 콘텐츠 원료 (장기요양.md 통합)

조합형 문장 생성에 바로 쓸 "공단이 인정하는 기록 기준". 추측 아닌 근거.

### 상담일지 (Indicator 11·12) — 1차 탭 핵심
- 필수 7필드: 상담일시, 수급자명, 상담방법(유선/방문), 상담직원명,
  **상담대상자명(관계)**, 상담내용, 상담결과
- ★대상자명+관계 필수: "박순자(본인)", "김OO 보호자(딸)" 형식 (단순 "수급자"=불인정)
- 상담직원 = 대표자/관리자(사회복지사) (일반 요양보호사면 리스크)
- 문자·카톡 상담 = 불인정 (유선/방문만)
- "특이사항 없음"·"지난달과 동일" 반복문구 = 불인정

### 직원 상담 (Indicator 2)
- 분기별 1회 이상. 대면/유선만. "조치 필요=예" 시 30일 내 조치 기록.

### 욕구사정 신규 (2026)
- 영양상태에 '기피식품 파악' 추가 / '연명의료결정 제도' 안내 신설

### 상태변화 품질
- 4주 연속 "변화 없음" = 형식적 기록 위험
- 신체·인지·정서·식사·수면·욕구·조치 중 최소 1개 구체적으로

### 시간 기록 (TOP1 실수)
- 총시간 = 세부활동 합계 (180 = 신체+인지+정서+가사)

## 11-A. 제품 형태 확정 (2026-06-19)

### 전달 방식: 모바일 웹 + 작성형(복사-붙여넣기)
- **모바일 웹** (폰 브라우저, 케어포와 다른 탭). 앱·크롬확장 아님.
  - 이유: 요양보호사·센터장이 현장/이동 중 폰으로 기록. 작성하다 막히면 같은 폰 다른 탭에서 참조.
- **작성형(B)** — "이렇게 쓰세요" 안내만이 아니라, **바로 복사해서 쓸 완성 문장 제공.**
  - 센터장이 이름만 바꿔 케어포에 붙여넣으면 끝. "막막함" 자체를 제거.

### 핵심 위험과 해결: 조합형 생성 (절대 잊지 말 것)
- ⚠️ **고정 예시 1개를 모두가 복붙하면 = 형식적 기록 = 평가 감점.**
  (네이버카페 근거: "상담내용이 짧거나 반복 문구로 보임"이 감점 사유)
- ✅ 해결: **상황을 탭으로 선택 → 변수 조합 → 매번 다른 문장 생성.**
  ```
  무엇이 변했나? [식사▾] 어떻게? [감소▾] 조치? [보호자 유선안내▾]
  → "식사량이 평소보다 감소하는 모습 관찰됨. 보호자에게 유선 안내,
     다음 방문 시 재확인 예정." → [복사]
  ```
  선택 조합이 다르면 문장도 다름. 같은 센터 안 수급자마다 다른 문장.
  복붙 편리함 + 자연스러움 동시 확보.

### 제품 흐름 (확정)
```
모바일 웹
 → 기록 항목 선택 (상태변화/급여계획/결과평가/상담일지)
 → 상황 탭 선택 (드롭다운·버튼)
 → 완성 문장 자동 생성 (조합형, 공단 근거 내장)
 → [복사하기] → 케어포에 붙여넣기
 → (선택) 다음 단계 연결 확인 = 검증(Level 1)
```

### ★★ UX 근본 구조: 비순차 · 모든 항목이 동등한 진입점 (2026-06-19 확정)

**현장의 진실: 일은 순차적으로 일어나지 않는다. 그래서 더 헷갈리고 더 까먹는다.**
- 오늘 김할머니 상태변화, 내일 이할아버지 상담일지, 모레 박할머니 결과평가 기한...
- 순서 없음. 그때그때 다름.

**따라서 "허브에서 가지로 뻗는 트리"는 틀렸다. 연쇄는 "거미줄(망)"이다.**
```
상태변화 ─┬→ 급여계획 변경
          └→ 결과평가 반영
급여계획 ←┬─ 욕구사정
          ├─ 상태변화
          └→ 설명·서명·통보
결과평가 ──→ 급여계획 재작성 → ...
```

**UX 구조:**
```
상단에 항상 탭이 떠 있음 (모든 항목 동등):
[상태변화][급여계획][결과평가][상담일지][욕구사정][위험도평가]...

센터장이 지금 케어포에서 쓰려는 게 뭐든
 → 그 탭을 누름 (어느 것이든, 아무 때나, 탭 하나로 즉시 도달)
 → 그 항목의 작성 가이드 + 조합형 문장 → 복사
 → ★ "이거 쓰면 OO도 해야 합니다" 연쇄 알림 (망에서 연결된 노드)
```

**핵심 원칙:**
- 순서를 강요하지 않는다. 어느 항목이든 동등한 진입점.
- 각 항목에서 출발해도, 망에서 연결된 다른 항목을 알려준다.
- "다양한 선택 환경에서 케어포에 쓸 때마다 참고가 되는" 구조.

### ★★★ 수급자 등록 (별칭 기반) — 연속성 추적의 핵심 (2026-06-19 확정)

**왜 필요한가:** 연쇄 알림(Level 2)이 작동하려면 "누구에게 무엇을 해야 하는지"
추적할 대상이 있어야 한다. 수급자마다 써야 할 항목·기한·진행 상태가 다르다.
```
등록 없음: "상태 악화 쓰면 계획 변경도 하세요" (추상적)
등록 있음: "나할머니는 지난주 상태 악화 기록했는데 계획 변경이 아직" (추적)
```

**신뢰와의 충돌 해결 — 별칭만 등록:**
- 등록하는 것: **별칭 1개**(센터장만 식별: "나할머니", "201호", "김O순") + 진행 상태(체크)
- 등록 안 하는 것: 실명, 주민번호, 주소, 등급, 진단명 등 진짜 개인정보
- 저장 위치: **센터장 폰 브라우저 안에만(localStorage). 서버 전송 0.**
- 안내: "수급자 개인정보를 수집하지 않습니다. 선생님이 알아볼 별칭만 기기에 저장됩니다."

**제품 완성 흐름:**
```
[수급자 보드] 별칭으로 등록된 수급자들 + 각자 "이번 달 해야 할 것"
  나할머니 → 욕구사정 기한 D-5, 상태변화 미작성
  가할아버지 → 결과평가 후 계획재작성 D-12
   ↓ 수급자 탭 선택
[작성 가이드] 그 수급자에게 지금 쓸 항목 선택 → 조합형 문장 → 복사 → 케어포
   ↓
[연쇄 알림] "이거 썼으면 OO도 하세요" + 그 수급자 보드에 자동 반영
```
수급자 보드 = "누가 무엇을 빠뜨렸나" 한눈에. 이게 연속성.

**기술 주의:** 지속을 위해 localStorage 필요(기존엔 안 썼음). Vercel 배포 앱은 작동.
"폰 바꾸면 사라짐" 한계는 "내보내기/불러오기"로 보완.

### ★ 알림 설계 (2026-06-19 확정) — "까먹지 않게"의 핵심

**"알림"에는 기술적으로 다른 두 가지가 섞여 있다. 구분해야 한다.**

| 종류 | 무엇 | 기술 | 신뢰 | 지금 가능? |
|---|---|---|---|---|
| 인앱 알림 | 앱 열면 수급자 보드에 "나할머니 D-5" 표시 | 쉬움(localStorage 날짜 계산) | 유지 | ✅ |
| 푸시 알림 | 앱 안 열어도 폰에 띵 (카톡처럼) | 모바일웹 거의 불가, 서버 필요 | 충돌 | ❌ |

**근본 긴장:** 푸시를 보내려면 서버가 "누가 무슨 기한인지" 알아야 함
→ "데이터를 서버에 안 보낸다 = 신뢰" 원칙과 정면충돌. 모바일웹은 푸시도 거의 안 됨.

**확정: 길 A — 인앱 알림 + 습관 유도 (지금)**
1. 푸시는 포기. 대신 앱 열면 수급자 보드에 강력하게 표시.
2. "케어포 쓸 때 우리 앱도 같이 연다" 습관 유도. 센터장은 케어포를 매일 엶.
3. 신뢰 유지 — 데이터 로컬에만, 푸시 서버 없음, "개인정보 안 나감" 100% 진실.
4. 가벼운 외부 리마인더는 별도 가능 — 카톡채널/문자로 "이번 주 점검하세요"
   (일반 문구, 개인정보 아님. "나할머니"가 아니라 "이번 주 점검").

**미래(검증 후):** 웹푸시(별칭만 서버, 안드로이드 위주) → 네이티브 앱(진짜 푸시).
지금은 지불의향 0명 단계 → 인앱 알림으로 가치부터 검증.

### 제품 특징
- 모바일 최적화 (큰 터치 버튼, 한 손 조작)
- 데이터 입력 최소 (선택지 탭만, 개인정보 입력 없음 → 신뢰)
- 원터치 복사 버튼
- 조합형 생성 (평가자 패턴 인식 방지)
- 공단 90+ 문서 근거 내장 (각 문장이 평가 기준 충족)

## 13-A. 화면 구조 (IA) 확정 (2026-06-19) — 3화면 + 홈

UX 리뷰 결과: 본인의 2화면(카테고리 가이드 + 수급자 보드)은 옳으나,
"오늘 뭘 해야 하나"에 답하는 **홈 화면이 빠져** 있었음. 그게 매일 여는 이유.

**실무자의 사고 축 = 시간(마감). 카테고리도 수급자도 아님.**
현장은 무질서 → 사람이 의지하는 단 하나의 축은 "언제까지".

### MVP 3화면 (관계가 핵심)
```
③ 수급자 보드(별칭 등록·진행 추적)  ← 데이터 공급원
        ↓ 시간순 계산
① 오늘의 보드 (홈) ★매일 여는 화면·제품 첫인상
   "이번 주 빠뜨리면 안 되는 것"
   🔴 나할머니 — 욕구사정 D-2
   🔴 가할아버지 — 결과평가 후 계획재작성 안 함
   🟡 다어르신 — 상태변화 3주째 없음
        ↓ 항목 누름
② 작성 가이드 (카테고리·조합형 문장) ← 막혔을 때
   조합형 문장 생성 → [복사] → 케어포
        ↓
   연쇄 알림 → ①에 새 할일 자동 추가 (루프)
```

### 포지셔닝 확정: "누락 방지" (검증 아님)
- 메인: "방문요양 기록, 빠뜨리지 않게"
- 부제: "이거 쓰면 저것도 — 놓치기 쉬운 연결을 챙겨드립니다"
- "감점 방지"는 결과로 부제에. "검증"은 약하니 버림.

### 화면 우선순위
1. 오늘의 보드(홈) — 하루에 가장 많이 봄, 첫인상, 반복 사용성의 열쇠
2. 작성 가이드 — 막혔을 때, 조합형 문장
3. 수급자 보드 — 연속성의 뿌리, ①의 데이터원
(4. 연쇄 알림은 ①②③에 내장 / 5. 월간 점검 리포트=검증, 평가 직전용, 나중)

### ★ 오늘의 보드 — 세 축 가중 정렬 (2026-06-19)

오늘의 보드에 보여줄 항목의 우선순위는 **한 축이 아니라 세 축의 조합**이다:
```
사용빈도: 상태변화(매일) > 상담(매주) > 급여계획(연간) > 결과평가(연간)
중요도:   급여계획(100) > 상태변화(90) > 상담(85) > 결과평가(80)
누락위험: 결과평가(최고) > 상담(몰아씀) > 상태변화 > 급여계획(보통 알림뜸)
```
세 축이 전부 다름. 한 축만 고르면 안 됨.
→ 위험도 점수 = 마감 임박도 × 중요도 × 누락 확률 로 가중 정렬.

**첫 화면 예시:**
```
🔴 나할머니 — 급여계획 만료 D-2 (중요도 100 × 임박 → 최상위)
🔴 가할아버지 — 결과평가 후 계획재작성 15일째 미완 (누락위험 최고)
🟡 다어르신 — 이번 달 방문상담 미완료 (상담 누락 → 가산금 영향)
🟡 라어르신 — 상태변화 기록 2주째 없음 (빈도 최고 항목 미작성)
```

### ★ 수급자 보드 — "한 사람의 타임라인" (2026-06-19)

케어포는 메뉴가 따로 놈 → 한 수급자의 흐름을 보려면 탭 4개를 열어야 함.
우리의 수급자 보드는 **"이 수급자에게 일어난 일의 시간순 흐름"**으로 보여줌:
```
나할머니 타임라인:
  06/01 욕구사정 실시 ✅
  06/03 급여제공계획 작성 ✅ → 공단 통보 ✅
  06/10 상태변화: 식사량 감소 ✅
  06/10 → [연쇄] 급여계획 변경 검토? ❌ 미완
  06/15 상담: ??  ❌ 미작성
  06/30 결과평가: ?? ⏳ 기한 내
```
체크리스트가 아니라 **흐름**. 이게 "기록의 연속성"을 눈으로 보는 것.
케어포가 절대 못 하는 것.

### ★ 작성 가이드 — 교차 참조 표시 (2026-06-19)

**진짜 공포:** "요양보호사가 상태변화에 '기침'이라 적었는데
나는 상담일지에 '건강하심'이라고 복붙" → 평가관 먹잇감.

해결: 상담일지 작성 가이드에서, **그 수급자의 최근 상태변화 기록을 옆에 표시.**
```
[상담일지 작성 — 나할머니]
📋 최근 상태변화 (참고): "06/10 식사량 감소, 보호자 유선 안내"
→ 상담 내용에 이 상태를 반영하세요.
  예: "식사량 감소 상태 지속 여부 확인, 보호자와 식사 보조 방법 상의"
```
이렇게 하면 **불일치를 사후 탐지가 아니라 사전 예방**함.
수급자 등록 + 타임라인이 있어야 이 교차 참조가 가능.
→ 수급자 등록이 "편의"가 아니라 "교차 참조의 전제 조건"임이 재확인됨.

## 13-B. 커스텀(개인 설정) — 어디까지 (2026-06-19)

센터장마다 보고 싶은 방식이 다름. 단, 검증 0명 단계이므로 **과하지 않게**.

**지금 넣을 커스텀 (가볍고 가치 큼):**
- 홈(오늘의 보드) 정렬: 마감순 / 수급자순 / 항목순 선택
- 알림 임박 기준: D-3 / D-5 / D-7 중 선택 (언제부터 빨갛게 뜰지)
- 탭 순서/표시: 자주 쓰는 기록 항목을 앞으로, 안 쓰는 항목 숨기기
- 수급자 보드: 별칭 자유 입력(본인 방식대로), 정렬 직접
- 모두 localStorage 저장 (개인 기기, 서버 0)

**지금 안 넣을 커스텀 (과잉):**
- 문장 템플릿 자체를 사용자가 편집 (공단 근거 깨질 위험)
- 색·폰트 테마 변경 (Section D 일관성 깨짐)
- 복잡한 대시보드 위젯 배치 (검증 전 과투자)

**원칙:** 커스텀은 "보는 방식"까지만. "내용(공단 근거 문장)"은 우리가 통제.
검증 후 사용자 요청 기반으로 확장.

## 13. 작업 원칙 (잊지 말 것)
- 새 정보 오면 strategic-lens 적용 (관성 금지, 엔진vs프레임 판단)
- 질문은 level10-upgrade (표면 질문 → 진짜 질문으로)
- 클로드 코드: 계획→구현→검증 분리. 구현자 모드 ≠ QA 모드.
- 핵심 판단은 본인이. Claude는 선택지+근거 제시.
- "이미 만든 게 아까워서"는 판단 기준 아님. "고객에게 가치인가"만.
- 엔진(14개 규칙)은 살아있다. 시점·표현·진입점만 바꾼다.
